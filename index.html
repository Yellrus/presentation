Вода

Инструменты, такие как препроцессоры и постпроцессоры, имеют большое значение для улучшения опыта разработки. Но одних только этих инструментов недостаточно для решения проблемы хранения большего количества CSS в проекте. Чтобы решить эту проблему, люди начали документировать разные рекомендации о том, как писать CSS, обычно называемые методологиями.

Если мы в идеальном мире

Предсказуемость

Предсказуемость для CSS означает, что ваши правила ведут себя ожидаемо. Когда вы добавляете или изменяете какое-то правило, оно не должно влиять на части сайта, которые вы не собирались менять. Это вряд ли необходимо маленьким сайтам, которые редко изменяются, но для больших сайтов с десятками или сотнями страниц предсказуемый CSS необходим.

Повторное использование

CSS-правила должны быть достаточно абстрактны и независимы, чтобы можно было быстро создать новые компоненты из существующих частей без необходимости что-то переписывать или снова бороться с уже решёнными проблемами.

Поддержка

Когда нужно добавить или изменить новые компоненты или возможности, это не должно приводить к рефакторингу существующего CSS. Добавление компонента А на страницу не должно своим появлением ломать компонент Б.

Масштабируемость

Растущий в размерах и сложности сайт обычно требует для поддержки большего числа разработчиков. Масштабируемость для CSS означает, что он с лёгкостью может поддерживаться как одним человеком, так и большой командой разработчиков. Также это значит, что архитектура CSS вашего сайта достаточно доступна и не требует чрезмерных усилий для понимания. То, что вы сегодня единственный разработчик, имеющий дело с CSS, не значит, что так будет всегда.

OOCSS (Object Oriented CSS)

Основан на двух принципах:

  1. Раздельная структура и оболочка (отделение "шкуры" от структуры)
  2. Это отдельный контейнер и контент(содержимое)

Пример:

			
				#widget {
					width: 200px;
					height: 50px;
					border: solid 1px #ccc;
					background: linear-gradient(#ccc, #222);
					box-shadow: rgba(0, 0, 0, .5) 2px 2px 5px;
				}

				#box {
					width: 400px;
					overflow: hidden;
					border: solid 1px #ccc;
					background: linear-gradient(#ccc, #222);
					box-shadow: rgba(0, 0, 0, .5) 2px 2px 5px;
				}

Пример:

				
					.box {
						width: 400px;
						overflow: hidden;
					}

					.widget {
						width: 500px;
						min-height: 200px;
						overflow: auto;
					}

					.skin {
						border: solid 1px #ccc;
						background: linear-gradient(#ccc, #222);
						box-shadow: rgba(0, 0, 0, .5) 2px 2px 5px;
					}

SMACSS (Scalable and Modular Architecture for CSS)

Разделение стилей на

  1. Base rules — базовые стили
  2. Layout rules — стили макета
  3. Modules rules — стили модулей
  4. State rules — стили состояния
  5. Theme rules — оформление

Пример cтруктуры css:

					
							@import ‘_fonts’;

							@import ‘_variables’;

							@import ‘_base’;

							@import ‘_layouts’;

							@import ‘_modules’;

							@import ‘_states’;

Atomic CSS — атомарный CSS

При использовании такого подхода для каждого повторно используемого свойства должен быть сформирован отдельный класс. Пример: стиль «margin-bottom: 10px» предполагает создание класса «mb-10», стиль «width: 200px» создание класса «w-200».

БЭМ

Блок Элемент Модификатор

Блок

Функционально независимый компонент страницы, который может быть повторно использован. В HTML блоки представлены атрибутом class.

Особенности:

  1. Название блока характеризует смысл («что это?» — «меню»: menu, «кнопка»: button), а не состояние («какой, как выглядит?» — «красный»: red, «большой»: big).
  2. Блок не должен влиять на свое окружение
  3. В CSS по БЭМ также не рекомендуется использовать селекторы по тегам или id.

Пример:

					<!-- Верно. Семантически осмысленный блок `error` -->
					<div class="error"></div>
					<!-- Неверно. Описывается внешний вид -->
					<div class="red-text"></div>

Возможности блоков:

- Блоки можно вкладывать друг в друга.

			<div class="page-header">
					<a class="logotype"></a>
					<div class="hero">
				 		<form class="search-form"></form>
					</div>
			</div>

Элемент

Составная часть блока, которая не может использоваться в отрыве от него.

Особенности:

  1. Название элемента характеризует смысл («что это?» — «пункт»: item, «текст»: text), а не состояние («какой, как выглядит?» — «красный»: red, «большой»: big).
  2. Структура полного имени элемента соответствует схеме: имя-блока__имя-элемента. Имя элемента отделяется от имени блока двумя подчеркиваниями (__).

Пример:

- Элемент не может находится вне своего блока.
- Элемент — необязательный компонент блока. Не у всех блоков должны быть элементы.

				<form class="search">
					<div class="search__inner">
						<input class="search__input"></input>
				 		<button class="search__btn btn" type="submit">Найти</button>
						</div>
					</div>
				</form>

Модификатор

Cущность, определяющая внешний вид, состояние или поведение блока либо элемента.

Особенности:

  1. Название модификатора характеризует внешний вид («какой размер?», «какая тема?» и т. п.), состояние, поведение
  2. Имя модификатора отделяется от имени блока или элемента одним подчеркиванием (_).
			 <button class="search__btn btn_disabled" type="submit">Найти</button>

Типы модификаторов:

  1. Булевый
  2. Ключ-значение
			 <button class="search__btn btn_disabled btn_size_xs btn_color_red btn_theme_newyear" type="submit">Найти</button>

В его примере:

			
				.media {}
				.media__img {}
				.media__img--rev {}
				.media__body {}

Почему БЕМ

  1. Позволяет создавать абсолютно независимые блоки. Блоки и элементы получают уникальные имена, так что стили для одного элемента ничего не поломают в другом;
  2. Помогает легко придумывать любое количество классов, не повторяющихся между собой;
  3. БЭМ помогает писать самодокументирующийся код, в классе любого элемента содержится информация о нём.

Что не дает БЕМ

  1. методология (никто не запретит вам сломать и нарушать);
  2. изоляция присутствует, если придерживаться правил (но вы работаете с людьми, а они ошибаются; легко сломать);
  3. WYSIWYG, работа с текстом (скорее всего ваш WYSIWYG надо будет настраивать или используйте каскад);

CSS в JS

Был представлен в 2014 году как методология, организованная вокруг определения стилей CSS не в отдельном .css файле, а непосредственно в каждом компоненте.

Цель методологии “CSS в JS”состоит в том, чтобы иметь возможность определять компоненты с жесткими границами, которые состоят из их собственного инкапсулированного HTML/CSS/JS, так что CSS в одном компоненте не имеет шансов повлиять на какие-либо другие компоненты.
